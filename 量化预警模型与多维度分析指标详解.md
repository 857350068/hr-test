## 一、核心设计思想

### 1.1 设计原则

本系统所有数据分析规则、统计口径、预警判定模型，均基于中小企业人力资源管理实际运营标准进行量化设定，摒弃模糊化定性描述。具体遵循以下原则：

1. **数据驱动原则**：所有预警判定必须基于实际业务数据，而非主观经验
2. **阈值可配置原则**：所有临界阈值支持后台管理员自定义配置，适配不同企业规模
3. **双库协同原则**：MySQL负责实时业务判定，Hive负责批量历史分析
4. **多因子综合原则**：预警模型采用多因子加权打分，避免单一指标误判
5. **分级预警原则**：区分高、中、低三级风险，对应不同处理策略

### 1.2 数据存储与计算分工

| 数据库 | 存储内容 | 计算职责 | 响应时效 |
|--------|---------|---------|---------|
| **MySQL 8.0** | 员工档案、每日考勤、月度薪资、绩效记录、岗位信息、请假异动 | 单人员、短周期的精细化规则判定 | 实时（毫秒级） |
| **Hive 3.1.3** | 多年历史HR全量数据（通过ETL从MySQL同步） | 长周期、跨部门、多维度的批量统计与趋势分析 | 批量（分钟级） |

### 1.3 ETL数据同步流程

```
每日凌晨定时任务执行流程：
┌─────────────────────────────────────────────────────────────┐
│  1. 抽取MySQL增量数据                                        │
│     - 昨日新增/修改的考勤记录                                 │
│     - 昨日新增/修改的薪资发放记录                             │
│     - 昨日新增/修改的绩效评估记录                             │
│     - 昨日新增/修改的员工异动记录                             │
├─────────────────────────────────────────────────────────────┤
│  2. 数据清洗                                                 │
│     - 过滤无效记录（deleted=1）                               │
│     - 去除重复数据                                           │
│     - 补充缺失字段（默认值填充）                               │
│     - 数据格式标准化                                         │
├─────────────────────────────────────────────────────────────┤
│  3. 加载至Hive数仓                                           │
│     - 追加到对应分区（dt='yyyy-MM-dd'）                       │
│     - 更新维度表（dim_employee）                              │
│     - 更新事实表（fact_attendance、fact_salary等）            │
└─────────────────────────────────────────────────────────────┘
```

## 三、六大类量化预警模型

### 3.1 员工个体：离职风险预警（核心预警）

#### 3.1.1 预警设计思想

从MySQL读取员工的**考勤、请假、绩效、薪资调整**等实际业务数据，采用多因子加权打分模型，综合判定离职风险。

#### 3.1.2 判定因子与具体阈值

| 序号 | 判定因子 | 数据来源表 | 具体阈值条件 | 权重 | 得分计算方式 |
|------|---------|-----------|-------------|------|-------------|
| 1 | **考勤异常** | `attendance` | 连续7天迟到/早退，或月度旷工≥2次 | 25% | 满足条件得1分，否则0分 |
| 2 | **请假频率** | `emp_leave` | 月度主动请假≥5天（40小时） | 20% | 满足条件得1分，否则0分 |
| 3 | **绩效表现** | `performance_evaluation` | 连续2个季度绩效评级为D（不合格） | 20% | 满足条件得1分，否则0分 |
| 4 | **薪资调整** | `salary_adjustment` | 近6个月无调薪，或调薪幅度<5% | 15% | 满足条件得1分，否则0分 |
| 5 | **工龄因素** | `employee` | 入职<6个月或3-5年（跳槽高发期） | 10% | 满足条件得1分，否则0分 |
| 6 | **学历因素** | `employee` | 本科/硕士/博士（跳槽倾向高） | 10% | 满足条件得1分，否则0分 |

#### 3.1.3 计算公式

```
离职风险得分 = 考勤异常得分 × 0.25 
            + 请假频率得分 × 0.20 
            + 绩效表现得分 × 0.20 
            + 薪资调整得分 × 0.15 
            + 工龄因素得分 × 0.10 
            + 学历因素得分 × 0.10
```

#### 3.1.4 触发规则

| 风险等级 | 得分范围 | 处理策略 |
|---------|---------|---------|
| **高风险** | 得分 ≥ 0.60 | 系统自动标记为高离职风险员工，推送预警给HR与部门负责人，建议安排面谈 |
| **中风险** | 0.40 ≤ 得分 < 0.60 | 纳入观察名单，定期关注其考勤、绩效变化 |
| **低风险** | 得分 < 0.40 | 正常状态，不触发预警 |

#### 3.1.5 SQL实现示例

```sql
-- 计算员工离职风险得分（MySQL实现）
SELECT 
  e.emp_id,
  e.emp_no,
  e.emp_name,
  e.department,
  e.position,
  
  -- 考勤异常得分（25%）
  CASE 
    WHEN (SELECT COUNT(*) FROM attendance 
          WHERE emp_id = e.emp_id AND attendance_type = 3 
          AND YEAR(attendance_date) = YEAR(CURRENT_DATE) 
          AND MONTH(attendance_date) = MONTH(CURRENT_DATE)) >= 2 
    THEN 0.25
    ELSE 0
  END AS attendance_score,
  
  -- 请假频率得分（20%）
  CASE 
    WHEN (SELECT COALESCE(SUM(leave_duration), 0) FROM emp_leave 
          WHERE emp_id = e.emp_id AND leave_type IN (0, 1) 
          AND approval_status = 1 
          AND YEAR(start_time) = YEAR(CURRENT_DATE) 
          AND MONTH(start_time) = MONTH(CURRENT_DATE)) >= 40 
    THEN 0.20
    ELSE 0
  END AS leave_score,
  
  -- 绩效表现得分（20%）
  CASE 
    WHEN (SELECT COUNT(*) FROM performance_evaluation 
          WHERE emp_id = e.emp_id AND performance_level = 'D' 
          AND evaluation_status = 3 
          ORDER BY year DESC, quarter DESC LIMIT 2) >= 2 
    THEN 0.20
    ELSE 0
  END AS performance_score,
  
  -- 薪资调整得分（15%）
  CASE 
    WHEN (SELECT COUNT(*) FROM salary_adjustment 
          WHERE emp_id = e.emp_id AND adjustment_type = 3 
          AND approval_status = 1 
          AND effective_date >= DATE_SUB(CURRENT_DATE, INTERVAL 6 MONTH)) = 0 
    THEN 0.15
    WHEN (SELECT AVG(adjustment_rate) FROM salary_adjustment 
          WHERE emp_id = e.emp_id AND adjustment_type = 3 
          AND approval_status = 1 
          AND effective_date >= DATE_SUB(CURRENT_DATE, INTERVAL 6 MONTH)) < 5 
    THEN 0.15
    ELSE 0
  END AS salary_score,
  
  -- 工龄因素得分（10%）
  CASE 
    WHEN DATEDIFF(CURRENT_DATE, e.hire_date) < 180 THEN 0.10
    WHEN DATEDIFF(CURRENT_DATE, e.hire_date) BETWEEN 1095 AND 1825 THEN 0.10
    ELSE 0
  END AS tenure_score,
  
  -- 学历因素得分（10%）
  CASE 
    WHEN e.education IN ('本科', '硕士', '博士') THEN 0.10
    ELSE 0
  END AS education_score
  
FROM employee e
WHERE e.status = 1 AND e.deleted = 0;
```

#### 3.1.6 风险因素说明生成

```java
// Java实现示例（TurnoverWarningService.java）
private List<String> getRiskFactors(Map<String, Object> emp, double riskScore) {
    List<String> factors = new ArrayList<>();
    
    // 考勤异常说明
    if (hasAttendanceIssue(emp)) {
        factors.add("考勤异常：连续迟到/早退或月度旷工≥2次");
    }
    
    // 请假频率说明
    if (hasHighLeaveFrequency(emp)) {
        factors.add("请假频繁：月度主动请假≥5天");
    }
    
    // 绩效表现说明
    if (hasPoorPerformance(emp)) {
        factors.add("绩效不佳：连续2个季度绩效评级为D");
    }
    
    // 薪资调整说明
    if (hasNoSalaryAdjustment(emp)) {
        factors.add("薪资停滞：近6个月无调薪或调薪幅度<5%");
    }
    
    // 工龄因素说明
    Integer tenureDays = (Integer) emp.get("tenure_days");
    if (tenureDays != null && tenureDays < 180) {
        factors.add("入职时间较短（<6个月）");
    } else if (tenureDays != null && tenureDays > 1095 && tenureDays < 1825) {
        factors.add("处于跳槽高发期（工龄3-5年）");
    }
    
    // 学历因素说明
    String education = (String) emp.get("education");
    if ("本科".equals(education) || "硕士".equals(education) || "博士".equals(education)) {
        factors.add("高学历人才（跳槽倾向高）");
    }
    
    return factors;
}
```

### 3.2 部门级：人员流失预警

#### 3.2.1 预警设计思想

面向部门整体，由Hive统计周期内汇总数据，当部门流失指标超出预设阈值时触发预警。

#### 3.2.2 判定条件

| 序号 | 预警类型 | 数据来源 | 具体阈值 | 触发条件 | 预警级别 |
|------|---------|---------|---------|---------|---------|
| 1 | **月度离职人数** | `dim_employee` | ≥3人 | 单部门月度`resign_date`非空人数≥3 | 高 |
| 2 | **月度流失率** | `dim_employee` | >15% | 月度离职人数/月初在职人数>15% | 高 |
| 3 | **核心岗位流失** | `dim_employee` | ≥1人 | `position`包含"工程师/经理/主管"且月度离职≥1人 | 中 |

#### 3.2.3 计算公式

```
月度流失率 = 月度离职人数 / 月初在职人数 × 100%
```

#### 3.2.4 触发规则

满足任意一条条件，即触发部门人才流失预警，推送至HR负责人和部门负责人。

#### 3.2.5 SQL实现示例

```sql
-- 部门流失预警查询（Hive实现）
SELECT 
  department,
  COUNT(CASE WHEN resign_date IS NOT NULL 
             AND YEAR(resign_date) = YEAR(CURRENT_DATE) 
             AND MONTH(resign_date) = MONTH(CURRENT_DATE) 
        THEN 1 END) AS monthly_resign_count,
  
  COUNT(CASE WHEN status = 1 THEN 1 END) AS current_active_count,
  
  COUNT(CASE WHEN resign_date IS NOT NULL 
             AND YEAR(resign_date) = YEAR(CURRENT_DATE) 
             AND MONTH(resign_date) = MONTH(CURRENT_DATE) 
        THEN 1 END) * 100.0 / 
  COUNT(CASE WHEN status = 1 THEN 1 END) AS monthly_turnover_rate,
  
  COUNT(CASE WHEN resign_date IS NOT NULL 
             AND position LIKE '%工程师%' 
             AND YEAR(resign_date) = YEAR(CURRENT_DATE) 
             AND MONTH(resign_date) = MONTH(CURRENT_DATE) 
        THEN 1 END) AS core_position_resign_count
  
FROM dim_employee
WHERE dt = '最新分区日期'
GROUP BY department
HAVING monthly_resign_count >= 3 
    OR monthly_turnover_rate > 15 
    OR core_position_resign_count >= 1;
```

### 3.3 人才缺口预警

#### 3.3.1 预警设计思想

基于招聘计划表（recruitment_plan），对比计划招聘人数与实际招聘人数，识别人才缺口。

#### 3.3.2 判定条件

| 序号 | 预警类型 | 数据来源 | 具体阈值 | 触发条件 | 预警级别 |
|------|---------|---------|---------|---------|---------|
| 1 | **编制缺口** | `recruitment_plan` | <80% | `hired_count/plan_count<0.8`且`status=1`（招聘中） | 高 |
| 2 | **核心岗位空缺** | `recruitment_plan` | >30天 | `position`包含"工程师/经理"且招聘中时长>30天 | 高 |
| 3 | **业务增长缺口** | `recruitment_plan` | 季度对比 | 季度业务量上涨但对应岗位`plan_count`未增加 | 中 |

#### 3.3.3 计算公式

```
编制满足率 = 已招聘人数 / 计划招聘人数 × 100%
招聘时长 = CURRENT_DATE - start_date
```

#### 3.3.4 SQL实现示例

```sql
-- 人才缺口预警查询
SELECT 
  department,
  position,
  plan_count,
  hired_count,
  hired_count * 100.0 / plan_count AS fill_rate,
  DATEDIFF(CURRENT_DATE, start_date) AS recruiting_days,
  
  CASE 
    WHEN hired_count * 100.0 / plan_count < 80 THEN '编制缺口'
    WHEN position LIKE '%工程师%' AND DATEDIFF(CURRENT_DATE, start_date) > 30 THEN '核心岗位空缺'
    ELSE '正常'
  END AS warning_type
  
FROM recruitment_plan
WHERE status = 1 
  AND (hired_count * 100.0 / plan_count < 80 
       OR (position LIKE '%工程师%' AND DATEDIFF(CURRENT_DATE, start_date) > 30));
```

### 3.4 人力成本超支预警

#### 3.4.1 预警设计思想

基于薪资发放表（salary_payment），监控人力成本的环比、同比增幅，以及占企业总成本的比例。

#### 3.4.2 判定条件

| 序号 | 预警类型 | 数据来源 | 具体阈值 | 触发条件 | 预警级别 |
|------|---------|---------|---------|---------|---------|
| 1 | **月度环比增幅** | `salary_payment` | >12% | 本月`SUM(total_gross_salary)`较上月增幅>12% | 高 |
| 2 | **年度成本占比** | `salary_payment` | >35% | 年度人力成本/企业总运营成本>35% | 高 |
| 3 | **部门连续超标** | `salary_payment` | 连续2个月 | 单部门薪酬开支连续2个月超预算 | 中 |

#### 3.4.3 计算公式

```
月度环比增幅 = (本月人力成本 - 上月人力成本) / 上月人力成本 × 100%
年度成本占比 = 年度人力成本 / 企业总运营成本 × 100%
```

#### 3.4.4 SQL实现示例

```sql
-- 月度人力成本环比增幅查询
SELECT 
  this_month.year,
  this_month.month,
  this_month.total AS this_month_cost,
  last_month.total AS last_month_cost,
  (this_month.total - last_month.total) / last_month.total * 100 AS growth_rate
  
FROM 
  (SELECT year, month, SUM(total_gross_salary) AS total 
   FROM salary_payment WHERE payment_status = 1 
   GROUP BY year, month) this_month,
   
  (SELECT year, month, SUM(total_gross_salary) AS total 
   FROM salary_payment WHERE payment_status = 1 
   GROUP BY year, month) last_month
   
WHERE this_month.year = YEAR(CURRENT_DATE) 
  AND this_month.month = MONTH(CURRENT_DATE)
  AND last_month.year = YEAR(DATE_SUB(CURRENT_DATE, INTERVAL 1 MONTH))
  AND last_month.month = MONTH(DATE_SUB(CURRENT_DATE, INTERVAL 1 MONTH))
  AND (this_month.total - last_month.total) / last_month.total * 100 > 12;
```

| 序号 | 预警类型 | 数据来源 | 具体阈值 | 触发条件 | 预警级别 |
|------|---------|---------|---------|---------|---------|

| 3 | **差评率高** | `training_enrollment` | >30% | 同一课程`feedback`差评率>30% | 中 |

#### 3.5.3 计算公式

```

```

#### 3.5.4 SQL实现示例

```sql

  COUNT(CASE WHEN te.attendance_status = 1 THEN 1 END) * 100.0 / COUNT(*) AS attendance_rate,
  AVG(te.score) AS avg_score
  
FROM training_enrollment te
JOIN employee e ON te.emp_id = e.emp_id
WHERE te.deleted = 0
GROUP BY e.department
HAVING attendance_rate < 60 OR avg_score < 65;
```

### 5.3 ETL同步流程详解

#### 5.3.1 同步频率

- **每日同步**：凌晨2:00执行，同步昨日增量数据
- **每周全量**：周日凌晨3:00执行，全量同步校验
- **实时同步**：关键业务数据（如员工离职）实时触发同步

#### 5.3.2 同步步骤

```
步骤1：抽取（Extract）
┌─────────────────────────────────────────────────────────────┐
│  从MySQL抽取昨日增量数据：                                    │
│  - 考勤记录：WHERE DATE(create_time) = CURRENT_DATE - 1      │
│  - 薪资发放：WHERE DATE(create_time) = CURRENT_DATE - 1      │
│  - 绩效评估：WHERE DATE(update_time) = CURRENT_DATE - 1      │
│  - 员工异动：WHERE DATE(update_time) = CURRENT_DATE - 1      │
└─────────────────────────────────────────────────────────────┘

步骤2：清洗（Transform）
┌─────────────────────────────────────────────────────────────┐
│  数据清洗规则：                                               │
│  - 过滤无效记录：WHERE deleted = 0                            │
│  - 去除重复数据：GROUP BY 主键                                │
│  - 补充缺失字段：COALESCE(field, default_value)              │
│  - 数据格式标准化：DATE_FORMAT、TRIM等                        │
│  - 业务逻辑转换：status=0→'离职'，status=1→'在职'            │
└─────────────────────────────────────────────────────────────┘

步骤3：加载（Load）
┌─────────────────────────────────────────────────────────────┐
│  加载至Hive数仓：                                             │
│  - 追加到对应分区：INSERT INTO table PARTITION(dt='yyyy-MM-dd')│
│  - 更新维度表：MERGE INTO dim_employee                        │
│  - 更新事实表：INSERT INTO fact_attendance                    │
│  - 更新统计表：INSERT INTO agg_department_stats              │
└─────────────────────────────────────────────────────────────┘
```

## 七、答辩背诵要点

### 7.1 核心要点（必背）

1. **所有规则全部有具体数字、阈值、判定条件**，不是空话
   - 例如："连续7天迟到"、"月度旷工≥2次"、"月度请假≥5天"

2. **个人短期风险、日常数据查MySQL**
   - 考勤、请假、绩效、薪资调整等实时业务数据
   - 毫秒级响应，支撑实时预警判定

3. **多年大数据、部门汇总、多维度分析靠Hive**
   - OLAP批量计算，分钟级响应
   - 支撑趋势分析、部门级预警

4. **用打分、比例、人数、占比这种实际参数做预警**
   - 不是抽象描述，而是具体数值
   - 例如："得分≥60分为高风险"、"流失率>15%触发预警"

5. **管理员后台可以改数值**，适配不同公司规模
   - `analysis_rule`表存储可配置阈值
   - `warning_model`表存储模型权重

### 7.2 常见问题回答模板

#### 问题1：部门多少人离职触发预警？

**回答**：我们设置了三个触发条件，满足任意一个即触发预警：
1. 单部门月度离职人数≥3人
2. 月度人员流失率>15%
3. 核心岗位（工程师、经理、主管）月度离职≥1人

这些阈值都存储在`analysis_rule`表中，管理员可以在后台调整。

#### 问题2：员工离职预警靠哪些实际数据判断？

**回答**：我们采用六因子加权打分模型，从MySQL读取员工的实际业务数据：
1. **考勤异常**（权重25%）：连续7天迟到/早退，或月度旷工≥2次
2. **请假频率**（权重20%）：月度主动请假≥5天（40小时）
3. **绩效表现**（权重20%）：连续2个季度绩效评级为D
4. **薪资调整**（权重15%）：近6个月无调薪或调薪幅度<5%
5. **工龄因素**（权重10%）：入职<6个月或3-5年（跳槽高发期）
6. **学历因素**（权重10%）：本科/硕士/博士（跳槽倾向高）

综合得分≥60分触发高风险预警。

#### 问题3：标准是什么？

**回答**：所有标准都是基于企业HR管理实践设定的：
- 考勤异常：连续7天是考虑工作周，月度旷工≥2次是严重违纪
- 请假频率：月度请假≥5天意味着出勤率<75%，影响工作
- 绩效表现：连续2个季度不合格说明能力或态度有问题
- 薪资调整：近6个月无调薪说明薪酬竞争力不足
- 工龄因素：入职<6个月是磨合期，3-5年是跳槽高发期
- 学历因素：高学历人才市场竞争力强，跳槽倾向高

这些标准都可以在后台调整，适配不同企业规模和行业特点。

#### 问题4：MySQL和Hive怎么分工？

**回答**：我们采用双库协同模式：
- **MySQL**：存储实时业务数据（考勤、薪资、绩效），负责单人员、短周期的精细化判定，毫秒级响应
- **Hive**：存储多年历史数据（通过ETL同步），负责跨年、跨部门、多维度的批量统计，分钟级响应

例如：员工个人离职风险判定查MySQL，部门月度流失率统计查Hive。

#### 问题5：ETL怎么保证数据一致性？

**回答**：我们采用三步ETL流程：
1. **抽取**：每日凌晨从MySQL抽取昨日增量数据
2. **清洗**：过滤无效记录、去除重复、补充缺失字段、格式标准化
3. **加载**：追加到Hive对应分区，更新维度表和事实表

同步后还会进行数据质量校验：记录数一致性、关键字段非空、数据范围合理性。
