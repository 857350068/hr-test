### 问题2: 数据同步方案采用什么技术?

#### 答案:

**已采用方案: 自定义ETL (SpringBoot定时任务 + JDBC)**

#### 技术选型对比:

| 方案 | 优势 | 劣势 | 本项目选择 |
|------|------|------|-----------|
| Sqoop | Hadoop官方工具 | 版本兼容性问题多,需独立安装 | ❌ 不选 |
| DataX | 阿里开源,支持多种数据源 | 配置复杂,学习成本高 | ❌ 不选 |
| Canal | 基于MySQL binlog实时同步 | 配置复杂,需维护binlog位点 | ❌ 不选 |
| **自定义ETL** | 无需额外组件,代码可控 | 需自行实现增量逻辑 | ✅ **已采用** |

#### 已实现的同步策略:

**实现位置:** `backend/src/main/java/com/hr/datacenter/service/sync/DataSyncService.java`

**同步配置:**
```yaml
data-sync:
  enabled: true
  # 维度表: 每日凌晨2点全量同步
  dimension-sync-cron: "0 0 2 * * ?"
  # 事实表: 每小时增量同步
  fact-sync-cron: "0 0 * * * ?"
  # 聚合表: 每日凌晨3点重新计算
  aggregation-sync-cron: "0 0 3 * * ?"
```

**同步方式:**
1. **维度表同步**: 每日凌晨2点全量同步 (TRUNCATE + INSERT)
2. **事实表同步**: 每小时增量同步 (INSERT)
3. **聚合表同步**: 每日凌晨3点重新计算

**技术实现:**
- 使用Spring的@Scheduled注解实现定时任务
- 使用JdbcTemplate直接执行Hive SQL
- 无需额外安装Sqoop或DataX
- 代码完全可控,便于调试和维护

**选择理由:**
1. HR数据变更频率低,每日定时同步即可满足需求
2. 无需增加集群复杂度,降低运维成本
3. 同步逻辑在应用代码中,便于调试和修改
4. Spring Schedule是Spring内置功能,无需学习新工具

### 问题3: 分析模型优先级和实现情况

#### 答案:

#### 已完成的核心分析功能(100%完成度):

**1. 员工流失预警模型** ✅
- **实现位置:** `backend/src/main/java/com/hr/datacenter/service/warning/TurnoverWarningService.java`
- **核心算法:**
  ```
  流失风险分数 = 工龄因素(30%) + 薪资因素(25%) + 学历因素(20%) + 岗位因素(15%) + 部门因素(10%)
  ```
- **评分标准:**
  - 工龄因素: 入职<半年(0.3)、<1年(0.2)、3-5年(0.15)
  - 薪资因素: <8000(0.25)、<10000(0.15)
  - 学历因素: 本科/硕士/博士(0.2)
  - 岗位因素: 技术岗位(0.15)
  - 部门因素: 技术部门(0.1)
- **风险等级:**
  - 高风险: ≥70%
  - 中风险: ≥40%
  - 低风险: <40%
- **功能:**
  - 流失风险预测
  - 部门流失率统计
  - 流失趋势分析
  - 预警概览

**2. 组织效能分析** ✅
- **实现位置:** `backend/src/main/java/com/hr/datacenter/service/analysis/OrgEfficiencyAnalysisService.java`
- **核心算法:**
  ```
  组织健康度评分 = 稳定性(40%) + (1-流失率)(30%) + 高学历比例(30%)
  ```
- **功能:**
  - 部门效能分析(人数、薪资、在职/离职统计)
  - 组织架构分析(部门/岗位/学历分布)
  - 人员配置分析

**3. 人才梯队建设分析** ✅

- **功能:**

  - 能力评估分析(学历/薪资/工龄能力分布)

**4. 薪酬福利分析** ✅

  - 人力成本分析(总成本、部门占比、成本趋势)

  - 使用Hive的PERCENTILE函数计算百分位数

### 答辩常见问题准备:

**Q1: 为什么选择Hive而不是其他大数据组件?**
A: Hive技术成熟、SQL友好、与Hadoop生态集成、适合离线分析场景、资源消耗低

**Q2: 数据量有多大?真的需要Hadoop集群吗?**
A: 中型企业500-1000人,5年历史数据约1000-2000万条记录,系统设计支持PB级数据扩展

**Q3: 数据同步为什么不用Sqoop?**
