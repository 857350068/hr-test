## 一、前置统一说明

### 1.1 双库计算分工原则

| 计算载体 | 计算范围 | 数据特征 | 响应时效 | 典型场景 |
|---------|---------|---------|---------|---------|
| **MySQL** | 单人员、短周期 | 实时明细数据、个体预警 | 毫秒级 | 员工离职风险评分、个人考勤统计 |
| **Hive** | 部门/公司、长周期 | 海量历史汇总、批量指标预警 | 分钟级 | 部门流失率、年度成本分析 |

### 1.2 阈值配置原则

- **所有阈值支持后台自定义配置**，适配不同企业规模
- 阈值存储在`analysis_rule`表和`warning_model`表
- 管理员可在"规则管理"和"模型管理"页面调整

### 1.3 公式书写规范

- 所有公式采用标准数学符号表示
- 变量使用下标表示（如W₁、W₂）
- 百分比统一使用×100%表示
- 分数使用分数线表示（非斜杠）

## 二、六大预警模型完整公式体系

### 2.1 员工个体离职风险预警模型（加权评分法）

#### 2.1.1 评价因子与权重分配

| 评价因子 | 权重（满分） | 扣分/加分规则 | 数据来源 |
|---------|------------|--------------|---------|
| **考勤异常** | 25分 | 月度迟到早退≥7次 或 旷工≥2次，满分扣完 | `attendance`表 |
| **绩效等级** | 25分 | 连续2季度绩效评级为D（不合格），满分扣完 | `performance_evaluation`表 |
| **请假异动** | 20分 | 月度事假/调休≥5天，满分扣完 | `emp_leave`表 |
| **合同到期** | 15分 | 3个月内合同到期且无续约登记，满分扣完 | `employee`表 |
| **岗位异动** | 15分 | 频繁调岗、咨询离职流程，满分扣完 | `salary_adjustment`表 |

#### 2.1.2 综合得分计算公式

**标准公式**：
```
S = W₁ + W₂ + W₃ + W₄ + W₅
```

**变量解释**：
- **S**：员工离职风险综合得分（满分100分）
- **W₁**：考勤异常扣分（满分25分）
- **W₂**：绩效异常扣分（满分25分）
- **W₃**：请假异动扣分（满分20分）
- **W₄**：合同到期风险扣分（满分15分）
- **W₅**：岗位异动风险扣分（满分15分）

#### 2.1.3 预警判定规则

| 得分范围 | 风险等级 | 处理策略 |
|---------|---------|---------|
| **S ≥ 60分** | 高离职风险预警 | 系统自动推送预警给HR与部门负责人，建议安排面谈 |
| **40 ≤ S < 60分** | 中度风险提示 | 纳入观察名单，定期关注其考勤、绩效变化 |
| **S < 40分** | 无离职风险 | 正常状态，不触发预警 |

#### 2.1.4 计算载体

- **数据存储**：员工每日/月度行为数据存MySQL
- **计算方式**：定时计算个人得分（每日凌晨或实时触发）
- **响应时效**：毫秒级

#### 2.1.5 SQL实现示例

```sql
-- 计算员工离职风险得分
SELECT 
  e.emp_id,
  e.emp_name,
  
  -- 考勤异常得分（满分25分）
  CASE 
    WHEN (SELECT COUNT(*) FROM attendance 
          WHERE emp_id = e.emp_id 
          AND attendance_type IN (1, 2) -- 迟到或早退
          AND YEAR(attendance_date) = YEAR(CURRENT_DATE) 
          AND MONTH(attendance_date) = MONTH(CURRENT_DATE)) >= 7 
    THEN 25
    WHEN (SELECT COUNT(*) FROM attendance 
          WHERE emp_id = e.emp_id 
          AND attendance_type = 3 -- 旷工
          AND YEAR(attendance_date) = YEAR(CURRENT_DATE) 
          AND MONTH(attendance_date) = MONTH(CURRENT_DATE)) >= 2 
    THEN 25
    ELSE 0
  END AS W1,
  
  -- 绩效异常得分（满分25分）
  CASE 
    WHEN (SELECT COUNT(*) FROM performance_evaluation 
          WHERE emp_id = e.emp_id 
          AND performance_level = 'D' 
          AND evaluation_status = 3 
          ORDER BY year DESC, quarter DESC LIMIT 2) >= 2 
    THEN 25
    ELSE 0
  END AS W2,
  
  -- 请假异动得分（满分20分）
  CASE 
    WHEN (SELECT COALESCE(SUM(leave_duration), 0) / 8 FROM emp_leave 
          WHERE emp_id = e.emp_id 
          AND leave_type IN (0, 1) -- 事假或病假
          AND approval_status = 1 
          AND YEAR(start_time) = YEAR(CURRENT_DATE) 
          AND MONTH(start_time) = MONTH(CURRENT_DATE)) >= 5 
    THEN 20
    ELSE 0
  END AS W3,
  
  -- 合同到期得分（满分15分）
  -- 假设合同周期为3年，计算距离合同到期天数
  CASE 
    WHEN DATEDIFF(DATE_ADD(e.hire_date, INTERVAL 3 YEAR), CURRENT_DATE) <= 90 
    THEN 15
    ELSE 0
  END AS W4,
  
  -- 岗位异动得分（满分15分）
  CASE 
    WHEN (SELECT COUNT(*) FROM salary_adjustment 
          WHERE emp_id = e.emp_id 
          AND adjustment_type = 1 -- 调岗
          AND approval_status = 1 
          AND effective_date >= DATE_SUB(CURRENT_DATE, INTERVAL 3 MONTH)) >= 2 
    THEN 15
    ELSE 0
  END AS W5
  
FROM employee e
WHERE e.status = 1 AND e.deleted = 0;
```

### 2.2 部门人员流失预警模型

#### 2.2.1 核心公式：部门人员流失率

**标准公式**：
```
部门流失率 = (周期内部门主动离职人数 / 部门周期期初在岗总人数) × 100%
```

**变量解释**：
- **周期内部门主动离职人数**：统计周期内该部门员工状态从"在职"变为"离职"的人数
- **部门周期期初在岗总人数**：统计周期开始时该部门的在职员工总数
- **部门流失率**：百分比形式，表示部门人员流失的严重程度

#### 2.2.2 双重触发条件（满足任意一条即预警）

| 触发条件 | 具体阈值 | 预警级别 | 说明 |
|---------|---------|---------|------|
| **量化人数** | 单部门月度主动离职人数 ≥ 3人 | 高 | 直接人数判定，简单直观 |
| **比率阈值** | 部门月度流失率 > 15% | 高 | 比率判定，适配不同规模部门 |
| **特殊规则** | 核心/技术/管理岗位月度流失 ≥ 1人 | 中 | 关键岗位单独监控 |

#### 2.2.3 计算载体

- **数据存储**：跨月度、年度历史人员数据同步至Hive
- **计算方式**：OLAP批量统计各部门流失率
- **响应时效**：分钟级（批量计算）

#### 2.2.4 SQL实现示例（Hive）

```sql
-- 部门人员流失率统计
SELECT 
  department,
  
  -- 月初在岗人数
  COUNT(CASE WHEN status = 1 
             AND hire_date < DATE_FORMAT(CURRENT_DATE, 'yyyy-MM-01')
        THEN 1 END) AS initial_count,
  
  -- 月度离职人数
  COUNT(CASE WHEN resign_date IS NOT NULL 
             AND YEAR(resign_date) = YEAR(CURRENT_DATE) 
             AND MONTH(resign_date) = MONTH(CURRENT_DATE) 
        THEN 1 END) AS resign_count,
  
  -- 月度流失率
  COUNT(CASE WHEN resign_date IS NOT NULL 
             AND YEAR(resign_date) = YEAR(CURRENT_DATE) 
             AND MONTH(resign_date) = MONTH(CURRENT_DATE) 
        THEN 1 END) * 100.0 / 
  COUNT(CASE WHEN status = 1 
             AND hire_date < DATE_FORMAT(CURRENT_DATE, 'yyyy-MM-01')
        THEN 1 END) AS turnover_rate,
  
  -- 核心岗位流失人数
  COUNT(CASE WHEN resign_date IS NOT NULL 
             AND position LIKE '%工程师%' 
             AND YEAR(resign_date) = YEAR(CURRENT_DATE) 
             AND MONTH(resign_date) = MONTH(CURRENT_DATE) 
        THEN 1 END) AS core_resign_count
  
FROM dim_employee
WHERE dt = '最新分区日期'
GROUP BY department
HAVING resign_count >= 3 
    OR turnover_rate > 15 
    OR core_resign_count >= 1;
```

### 2.3 人才缺口预警模型

#### 2.3.1 核心公式：岗位编制饱和度

**标准公式**：
```
编制饱和度 = (部门实际在岗人数 / 部门核定编制人数) × 100%
```

**变量解释**：
- **部门实际在岗人数**：当前该部门实际在职的员工数量
- **部门核定编制人数**：企业核定该部门的标准编制人数
- **编制饱和度**：百分比形式，表示部门人员配置的充足程度

#### 2.3.2 预警触发条件

| 触发条件 | 具体阈值 | 预警级别 | 说明 |
|---------|---------|---------|------|
| **编制缺口** | 部门编制饱和度 < 80% | 高 | 人员严重不足 |
| **核心岗位空缺** | 核心岗位空缺时长 > 30天未完成招聘补员 | 高 | 关键岗位长期空缺 |
| **业务增长缺口** | 业务量上涨但在岗人员连续缩减 | 中 | 业务与人员不匹配 |

#### 2.3.3 计算载体

- **明细数据**：编制、岗位空缺明细在MySQL
- **批量统计**：全公司多部门饱和度批量统计由Hive完成
- **响应时效**：明细查询毫秒级，批量统计分钟级

#### 2.3.4 SQL实现示例

```sql
-- 人才缺口预警查询
SELECT 
  rp.department,
  rp.position,
  rp.plan_count AS authorized_count,  -- 核定编制人数
  COUNT(e.emp_id) AS actual_count,    -- 实际在岗人数
  COUNT(e.emp_id) * 100.0 / rp.plan_count AS saturation_rate,  -- 编制饱和度
  DATEDIFF(CURRENT_DATE, rp.start_date) AS vacancy_days  -- 空缺时长
  
FROM recruitment_plan rp
LEFT JOIN employee e ON rp.department = e.department 
                     AND rp.position = e.position 
                     AND e.status = 1
WHERE rp.status = 1  -- 招聘中
GROUP BY rp.department, rp.position, rp.plan_count, rp.start_date
HAVING saturation_rate < 80 
    OR (rp.position LIKE '%工程师%' AND DATEDIFF(CURRENT_DATE, rp.start_date) > 30);
```

### 2.4 人力成本超支预警模型

#### 2.4.1 公式1：人力成本环比增长率

**标准公式**：
```
成本环比增幅 = ((本月人力成本 - 上月人力成本) / 上月人力成本) × 100%
```

**变量解释**：
- **本月人力成本**：本月企业人力总成本（薪资总额+社保公积金+其他福利）
- **上月人力成本**：上月企业人力总成本
- **成本环比增幅**：百分比形式，表示人力成本的月度增长速度

#### 2.4.2 公式2：人力成本占比

**标准公式**：
```
人力成本占比 = (企业周期人力总成本 / 企业周期运营总成本) × 100%
```

**变量解释**：
- **企业周期人力总成本**：统计周期内企业人力总成本
- **企业周期运营总成本**：统计周期内企业运营总成本（含人力、物料、运营等）
- **人力成本占比**：百分比形式，表示人力成本在企业总成本中的比重

#### 2.4.3 预警触发条件

| 触发条件 | 具体阈值 | 预警级别 | 说明 |
|---------|---------|---------|------|
| **月度环比增幅** | 月度人力成本环比增幅 > 12% | 高 | 成本增长过快 |
| **年度成本占比** | 年度人力成本占比 > 35% | 高 | 人力成本占比过高 |
| **部门连续超标** | 单部门人力开支连续两月超出预算 | 中 | 部门成本失控 |

#### 2.4.4 计算载体

- **日常成本明细**：MySQL存储月度薪资成本明细
- **年度跨期分析**：Hive完成跨年、全公司成本汇总分析
- **响应时效**：明细查询毫秒级，年度分析分钟级

#### 2.4.5 SQL实现示例

```sql
-- 人力成本环比增幅计算
SELECT 
  this_month.year,
  this_month.month,
  this_month.total AS this_month_cost,
  last_month.total AS last_month_cost,
  (this_month.total - last_month.total) / last_month.total * 100 AS growth_rate
  
FROM 
  (SELECT year, month, SUM(total_gross_salary) AS total 
   FROM salary_payment 
   WHERE payment_status = 1 
   GROUP BY year, month) this_month,
   
  (SELECT year, month, SUM(total_gross_salary) AS total 
   FROM salary_payment 
   WHERE payment_status = 1 
   GROUP BY year, month) last_month
   
WHERE this_month.year = YEAR(CURRENT_DATE) 
  AND this_month.month = MONTH(CURRENT_DATE)
  AND last_month.year = YEAR(DATE_SUB(CURRENT_DATE, INTERVAL 1 MONTH))
  AND last_month.month = MONTH(DATE_SUB(CURRENT_DATE, INTERVAL 1 MONTH))
  AND (this_month.total - last_month.total) / last_month.total * 100 > 12;
```

### 2.5 培训效果异常预警模型

#### 2.5.1 公式1：部门培训参训率

**标准公式**：
```
参训率 = (实际参训员工数 / 应参训员工总数) × 100%
```

**变量解释**：
- **实际参训员工数**：实际参加培训的员工数量
- **应参训员工总数**：应该参加培训的员工总数（通常为部门在职人数）
- **参训率**：百分比形式，表示培训的覆盖程度

#### 2.5.2 公式2：培训考核合格率

**标准公式**：
```
合格率 = (考核合格及以上人数 / 参训总人数) × 100%
```

**变量解释**：
- **考核合格及以上人数**：培训考核成绩达到合格标准的人数
- **参训总人数**：实际参加培训的总人数
- **合格率**：百分比形式，表示培训效果的质量

#### 2.5.3 预警触发条件

| 触发条件 | 具体阈值 | 预警级别 | 说明 |
|---------|---------|---------|------|
| **参训率低** | 部门培训参训率 < 60% | 中 | 培训覆盖不足 |
| **考核成绩差** | 培训考核平均分 < 65分 或 合格率 < 70% | 高 | 培训效果不佳 |
| **差评率高** | 同一培训课程差评占比过高 | 中 | 培训质量有问题 |

#### 2.5.4 计算载体

- **数据存储**：培训报名、考核成绩存MySQL
- **计算方式**：MySQL+Hive协同（明细查询+批量统计）
- **响应时效**：毫秒级（明细）+分钟级（批量）

#### 2.5.5 SQL实现示例

```sql
-- 培训效果异常预警查询
SELECT 
  e.department,
  tc.course_name,
  
  -- 应参训人数
  COUNT(DISTINCT e.emp_id) AS should_attend_count,
  
  -- 实际参训人数
  COUNT(DISTINCT CASE WHEN te.attendance_status = 1 THEN te.emp_id END) AS actual_attend_count,
  
  -- 参训率
  COUNT(DISTINCT CASE WHEN te.attendance_status = 1 THEN te.emp_id END) * 100.0 / 
  COUNT(DISTINCT e.emp_id) AS attendance_rate,
  
  -- 考核平均分
  AVG(te.score) AS avg_score,
  
  -- 合格率
  COUNT(CASE WHEN te.score >= 60 THEN 1 END) * 100.0 / 
  COUNT(CASE WHEN te.score IS NOT NULL THEN 1 END) AS pass_rate
  
FROM training_course tc
JOIN training_enrollment te ON tc.course_id = te.course_id
JOIN employee e ON te.emp_id = e.emp_id
WHERE e.status = 1
GROUP BY e.department, tc.course_name
HAVING attendance_rate < 60 
    OR avg_score < 65 
    OR pass_rate < 70;
```

### 2.6 绩效异常预警模型

#### 2.6.1 核心公式：部门绩效不合格占比

**标准公式**：
```
绩效不合格占比 = (部门绩效不合格人数 / 部门总考核人数) × 100%
```

**变量解释**：
- **部门绩效不合格人数**：部门内绩效评级为D（不合格）的员工数量
- **部门总考核人数**：部门内参与绩效考核的总人数
- **绩效不合格占比**：百分比形式，表示部门绩效问题的严重程度

#### 2.6.2 预警触发条件

| 触发条件 | 具体阈值 | 预警级别 | 说明 |
|---------|---------|---------|------|
| **部门不合格率高** | 部门连续两季度绩效不合格占比 > 20% | 高 | 团队绩效问题严重 |
| **核心岗位垫底** | 关键岗位员工长期绩效垫底，无改进记录 | 中 | 关键人员绩效问题 |

#### 2.6.3 计算载体

- **数据存储**：绩效评估记录存MySQL
- **计算方式**：MySQL+Hive协同（明细查询+批量统计）
- **响应时效**：毫秒级（明细）+分钟级（批量）

#### 2.6.4 SQL实现示例

```sql
-- 绩效异常预警查询
SELECT 
  e.department,
  pe.year,
  pe.quarter,
  
  -- 总考核人数
  COUNT(*) AS total_count,
  
  -- 不合格人数
  COUNT(CASE WHEN pe.performance_level = 'D' THEN 1 END) AS poor_count,
  
  -- 不合格占比
  COUNT(CASE WHEN pe.performance_level = 'D' THEN 1 END) * 100.0 / COUNT(*) AS poor_rate
  
FROM performance_evaluation pe
JOIN employee e ON pe.emp_id = e.emp_id
WHERE pe.evaluation_status = 3  -- 已完成
  AND pe.period_type = 2  -- 季度评估
GROUP BY e.department, pe.year, pe.quarter
HAVING poor_rate > 20
ORDER BY e.department, pe.year DESC, pe.quarter DESC;
```

## 三、多维度分析核心指标公式

### 3.1 员工留存率

**标准公式**：
```
员工留存率 = (周期期末在职人数 / 周期期初入职总人数) × 100%
```

**变量解释**：
- **周期期末在职人数**：统计周期结束时仍在职的员工数量
- **周期期初入职总人数**：统计周期开始时入职的员工总数
- **员工留存率**：百分比形式，表示员工的稳定性

**应用场景**：新员工留存率分析、人才稳定性分析

---

### 3.2 人均人力成本

**标准公式**：
```
人均成本 = 部门周期人力总成本 / 部门平均在岗人数
```

**变量解释**：
- **部门周期人力总成本**：统计周期内部门人力总成本
- **部门平均在岗人数**：统计周期内部门平均在岗人数
- **人均成本**：元/人，表示部门人均人力成本

**应用场景**：薪酬竞争力分析、成本效率分析

---

### 3.3 部门人均产出

**标准公式**：
```
部门人均产出 = 部门业务收入 / 部门在职人数
```

**变量解释**：
- **部门业务收入**：部门创造的业务收入
- **部门在职人数**：部门当前在职员工数量
- **部门人均产出**：元/人，表示部门整体产出效率

**应用场景**：组织效能分析、部门效率对比

---

### 3.4 人员编制饱和度

**标准公式**：
```
编制饱和度 = (实际在岗人数 / 核定编制人数) × 100%
```

**变量解释**：
- **实际在岗人数**：当前实际在职的员工数量
- **核定编制人数**：企业核定的标准编制人数
- **编制饱和度**：百分比形式，表示人员配置的充足程度

**应用场景**：人才缺口分析、编制管理

---

### 3.5 在岗人数波动率

**标准公式**：
```
在岗人数波动率 = ((本月人数 - 上月人数) / 上月人数) × 100%
```

**变量解释**：
- **本月人数**：本月在职员工数量
- **上月人数**：上月在职员工数量
- **在岗人数波动率**：百分比形式，表示人员流动趋势

**应用场景**：组织稳定性分析、人员流动趋势监控

### 5.2 常见问题回答模板

#### 问题1：员工离职风险怎么计算的？

**回答**：我们采用五因子加权评分模型，计算公式是S = W₁ + W₂ + W₃ + W₄ + W₅，其中：
- W₁是考勤异常扣分（满分25分），月度迟到早退≥7次或旷工≥2次扣满分
- W₂是绩效异常扣分（满分25分），连续2季度绩效不合格扣满分
- W₃是请假异动扣分（满分20分），月度事假≥5天扣满分
- W₄是合同到期扣分（满分15分），3个月内合同到期扣满分
- W₅是岗位异动扣分（满分15分），频繁调岗扣满分

综合得分≥60分触发高离职风险预警，这些数据都从MySQL实时读取。

---

#### 问题2：部门流失率怎么算的？

**回答**：部门流失率 = 周期内主动离职人数 / 部门期初在岗总人数 × 100%。我们设置了三个触发条件，满足任意一个就预警：
1. 单部门月度主动离职人数 ≥ 3人
2. 部门月度流失率 > 15%
3. 核心技术/管理岗位月度流失 ≥ 1人

这些数据通过ETL同步到Hive，利用OLAP批量计算各部门流失率。

---

#### 问题3：MySQL和Hive怎么分工的？

**回答**：我们采用双库协同模式：
- **MySQL**负责单人员、短周期的实时明细数据计算，比如员工个人离职风险评分、个人考勤统计，响应时效是毫秒级
- **Hive**负责部门/公司级、长周期的海量历史汇总分析，比如部门流失率、年度成本分析，利用OLAP分布式计算能力，响应时效是分钟级

这样既保证了实时性，又支撑了海量数据分析。

---

#### 问题4：阈值可以调整吗？

**回答**：可以。所有阈值都存储在`analysis_rule`表和`warning_model`表中，管理员可以在后台的"规则管理"和"模型管理"页面调整。比如：
- 离职风险的高风险阈值默认是60分，可以调整为50分或70分
- 部门流失率阈值默认是15%，可以调整为10%或20%
- 成本环比增幅阈值默认是12%，可以调整为10%或15%

这样适配不同企业规模和行业特点。

---

#### 问题5：这些公式有什么实际意义？

**回答**：所有公式都基于企业HR管理实践：
- **离职风险公式**：考勤异常、绩效不合格、请假频繁都是离职前兆，通过量化打分提前预警
- **流失率公式**：15%是行业平均流失率，超过说明部门有问题
- **编制饱和度公式**：80%是人员配置的警戒线，低于说明严重缺人
- **成本增幅公式**：12%是成本增长的合理上限，超过说明成本失控
- **参训率公式**：60%是培训覆盖的基本要求，低于说明培训不到位
- **绩效不合格率公式**：20%是团队绩效问题的警戒线，超过说明团队管理有问题

这些都有明确的业务含义，不是随意设定的。
