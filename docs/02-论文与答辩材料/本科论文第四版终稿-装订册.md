## 附五 中英文摘要

### 中文摘要

随着企业数字化转型从“流程信息化”向“决策智能化”演进，人力资源管理系统的建设重点也从单一业务处理逐渐转向数据治理与决策支持。传统人事系统在员工档案、考勤、薪酬、培训、绩效等基础业务方面已较成熟，但在跨模块数据整合、统一口径分析、风险预警和报表治理方面仍存在明显短板，具体表现为数据孤岛严重、统计口径不一致、分析时效低、风险识别滞后、管理闭环不完整。针对上述问题，本文结合真实工程项目，设计并实现了一套基于 Hive 的人力资源数据中心系统。系统采用前后端分离架构，后端以 Spring Boot、MyBatis-Plus、Spring Security 与 JWT 为核心技术，前端基于 Vue3、Element Plus 与 ECharts 完成交互与可视化；数据层采用 MySQL 与 Hive（虚拟机兼容模式下支持 MySQL 分析链路）双数据源方案，实现“业务数据沉淀 + 分析服务输出 + 治理能力闭环”。系统在功能上覆盖员工、考勤、请假、绩效、薪酬、培训、招聘等业务模块，并在此基础上实现组织效能分析、人才梯队分析、薪酬结构分析以及流失、人才缺口、成本超支三类预警；在工程能力上实现了报表任务调度、导出与分享、消息通知、运行模式识别、脚本化部署与接口自动化验收。本文详细论述了系统需求分析、总体架构、数据库与接口设计、关键模块实现、部署与测试过程，并给出系统运行数据与理论设计参数的对比分析，解释误差来源与改进策略。经虚拟机集群主节点部署和全链路回归验证，系统能够稳定运行并满足毕业设计任务书指标要求。本文成果对中小型组织建设人力资源数据中心具有较强的参考价值与工程实践意义。

**关键词**：人力资源数据中心；Hive；Spring Boot；Vue3；数据治理；风险预警

### English Abstract

With the evolution of enterprise digital transformation from process informatization to decision intelligence, the focus of human resource information systems has shifted from transaction management to data governance and decision support. Although traditional HR systems are relatively mature in basic business functions such as employee records, attendance, salary, training, and performance, they still show significant weaknesses in cross-module data integration, unified analytical metrics, risk warning, and report governance. Typical problems include severe data silos, inconsistent statistical standards, low analytical timeliness, delayed risk identification, and incomplete management loops.

To address these issues, this thesis designs and implements a Hive-based Human Resource Data Center system based on a real engineering project. The system adopts a front-end and back-end separated architecture. The back-end is built with Spring Boot, MyBatis-Plus, Spring Security, and JWT, while the front-end is implemented using Vue3, Element Plus, and ECharts for interaction and visualization. At the data layer, a dual-source strategy is used: MySQL for transactional business data and Hive for analytical workloads (with a MySQL-compatible analysis mode in VM demo environments), forming a complete chain of business data accumulation, analytical service output, and governance loop.

Functionally, the system covers major HR domains including employee, attendance, leave, performance, salary, training, and recruitment. Based on these domains, it further provides organization efficiency analysis, talent pipeline analysis, salary structure analysis, and three risk warning modules (turnover risk, talent gap risk, and labor cost overrun risk). In terms of engineering capability, the system supports report scheduling, export and sharing, message notification, runtime profile identification, script-based deployment, and automated API acceptance verification.  

Deployment and regression verification on the VM cluster master node show that the system can run stably and meet the indicators defined in the graduation project task book. The results provide practical reference value for small and medium-sized organizations building HR data center platforms.

**Key Words**: Human Resource Data Center; Hive; Spring Boot; Vue3; Data Governance; Risk Warning

## 附七 正文

# 第一章 绪论

## 1.1 研究背景

在人力资源管理信息化发展过程中，系统建设重点长期集中于“流程电子化”与“业务可追踪”。这类系统在员工信息维护、考勤登记、请假审批、薪酬发放等事务层面已具有较高成熟度，但随着组织规模扩大和管理复杂度提升，仅靠事务系统已难以满足现代企业的管理需求。管理层需要的不仅是“记录结果”，更是“洞察规律、识别风险、支持决策”。

实践中常见问题包括：业务数据分散在多个模块、跨模块统计口径不一致、人工报表效率低、风险识别滞后、治理动作与分析结果脱节。以员工流失为例，若仅在离职发生后统计，很难形成事前预警；若缺少绩效、薪酬、任职时长等多维数据联动，预警也难以具有解释力。

因此，建设面向管理决策的人力资源数据中心成为必然方向。数据中心并不是简单的数据汇总库，而是一套涵盖数据沉淀、分析服务、预警治理、报表协同和过程留痕的系统工程。

## 1.2 研究目的与意义

### 1.2.1 研究目的

本文围绕真实工程项目，设计并实现一套基于 Hive 的人力资源数据中心系统，目标如下：

1. 打通员工、考勤、请假、绩效、薪酬、培训、招聘等核心业务数据链路；
2. 建立组织效能、人才梯队、薪酬结构等分析能力；
3. 建立流失、人才缺口、成本超支的预警能力；
4. 建立规则管理、模型管理、报表导出与分享、消息通知等治理能力；
5. 在虚拟机集群主节点环境完成可复现部署与测试闭环。

### 1.2.2 研究意义

从理论层面看，本课题将数据仓库、ETL、风险预警、RBAC 权限等方法整合应用于 HR 场景，形成可解释、可治理的数据中心实践。  
从应用层面看，系统可提升管理效率、减少信息孤岛、增强决策支持能力。  
从工程层面看，本课题沉淀了可部署、可回归、可演示的完整交付链路，具备较强实践参考价值。

## 1.3 国内外研究现状

国外在 People Analytics 方面起步较早，强调通过多源数据融合进行组织行为分析、员工流失预测、人才供应链评估等。典型特点包括：统一指标体系、可解释分析模型、与业务策略闭环联动。  

国内在人力资源信息化方面发展迅速，但中小组织普遍面临“系统多、分析弱、治理散”的现实问题。大量系统停留在事务处理层，分析模块多为静态报表，缺少模型管理和预警联动，且部署与运维规范化不足，导致系统难以长期稳定发挥决策价值。

基于此，本文重点关注“可落地”与“可复现”两项能力：既重视分析预警功能，也重视部署验收和工程稳定性。

## 1.4 研究内容与技术路线

本文研究内容包括：

- 需求分析与关键方案论证；
- 前后端分离 + 双数据源架构设计；
- 业务、分析、预警、报表、治理模块实现；
- VM 集群主节点部署与全链路回归；
- 运行数据与理论参数对比及误差分析。

技术路线为：

业务数据沉淀 -> 规则模型配置 -> 分析预警输出 -> 报表协同治理 -> 自动化验证。

# 第二章 相关理论与关键技术

## 2.1 人力资源数据中心理论基础

人力资源数据中心建设强调“业务数据可管理、分析数据可计算、治理过程可追溯”。其核心逻辑是从事务处理走向决策支持。为实现这一目标，系统必须实现三个统一：数据口径统一、指标定义统一、分析解释统一。只有这样，管理层才能基于同一事实进行决策，避免“同一问题多个答案”。

## 2.2 数据仓库与 ETL 理论

数据仓库理论强调面向主题、集成、稳定和时变。HR 领域中的主题通常包括人员、组织、绩效、成本、培训等。ETL 的作用在于将分散业务数据加工为可分析数据，降低统计口径偏差。  

本系统采用 MySQL + Hive 双数据源策略，标准模式使用 Hive 分析链路；兼容模式可退化为 MySQL 分析链路，保障演示场景可运行。该策略体现了“目标导向 + 工程务实”的设计理念。

## 2.3 预警分析方法基础

预警系统通常由指标、规则、模型三部分组成。  
指标负责表达业务状态，规则负责定义风险边界，模型负责刻画多因素影响关系。  

本项目实现了规则管理和模型管理，使预警逻辑由硬编码演进为可配置模式。该机制支持后续持续优化，也便于论文中论证系统可扩展性。

## 2.4 系统实现关键技术

1. Spring Boot：统一后端开发框架；  
2. MyBatis-Plus：简化持久层开发；  
3. Spring Security + JWT：无状态认证授权；  
4. Vue3 + Element Plus + ECharts：前端可视化实现；  
5. Vite：前端构建与开发代理；  
6. 脚本化部署：PowerShell + Shell 实现自动化部署。

## 2.5 本章小结

本章明确了系统建设的理论依据和技术支撑，为后续需求、设计与实现章节提供方法基础。

# 第三章 需求分析与方案论证

## 3.1 业务需求分析

### 3.1.1 核心业务需求

系统需覆盖员工管理、考勤管理、请假管理、绩效管理、薪酬管理、培训管理、招聘管理等模块，形成数据中心上游稳定数据来源。

### 3.1.2 决策分析需求

系统需支持组织效能分析、人才梯队分析、薪酬分析、风险预警，并支持图表可视化、筛选对比和结果导出。

### 3.1.3 治理协同需求

系统需支持规则管理、模型管理、报表任务管理、执行与分享日志、消息通知、收藏中心、操作日志，以保证分析结果可落地、可留痕、可审计。

## 3.2 非功能需求分析

1. 安全性：身份认证与权限控制；  
2. 稳定性：部署可重复、运行可恢复；  
3. 可维护性：模块分层、接口规范化；  
4. 可扩展性：规则模型可配置、模块可新增。

## 3.3 方案设计与比较

### 3.3.1 前后端架构方案

采用前后端分离架构，提升迭代效率与可扩展性。

### 3.3.2 数据架构方案

采用 MySQL + Hive 双数据源，兼顾事务处理和分析能力；在 VM 演示模式下允许兼容链路，提升部署成功率。

### 3.3.3 认证授权方案

采用 JWT + 角色权限控制，实现无状态认证和接口级授权。

## 3.4 可行性分析

技术上可行：技术栈成熟、社区支持完善；  
环境上可行：具备虚拟机与数据库环境；  
工程上可行：已有脚本化部署与自动化测试能力。

## 3.5 本章小结

本章明确了系统应做什么、为何这样做、如何可行，为总体设计奠定基础。

# 第四章 系统总体设计

## 4.1 架构设计

系统分为表示层、业务层、数据层、部署层。  
表示层负责可视化与交互；业务层负责逻辑计算与权限控制；数据层负责事务与分析；部署层负责统一访问与运行维护。

## 4.2 功能模块设计

模块分为五大类：

1. 认证与权限；
2. 业务数据源；
3. 数据分析与预警；
4. 报表与协同；
5. 数据治理。

## 4.3 数据库设计

### 4.3.1 关键表

- 用户权限类：`sys_user`、`sys_role`  
- 消息日志类：`sys_message`、`sys_operation_log`  
- 业务事实类：`employee`、`attendance`、`emp_leave`、`performance_goal`、`performance_evaluation`、`salary_payment`、`salary_adjustment`、`training_course`、`training_enrollment`、`recruitment_plan`  
- 治理类：`analysis_rule`、`warning_model`、`data_category`  
- 报表类：`report_task`、`report_execution_log`、`report_share_log`

### 4.3.2 约束设计

统一逻辑删除、创建更新时间、角色编码、任务日志追踪等机制，保障数据可治理与可追溯。

## 4.4 安全与权限设计

采用“后端强控制 + 前端弱可见”的双层策略：  
后端通过接口鉴权保障安全边界；前端通过菜单与路由控制优化用户体验。

## 4.5 接口与交互设计

接口统一业务返回结构，前端统一处理成功与异常反馈。  
导出接口独立处理 `blob` 流，避免与普通 JSON 接口冲突。

## 4.6 本章小结

本章完成系统蓝图设计，为实现与测试提供清晰边界和执行标准。

# 第五章 系统详细实现

## 5.1 后端实现

### 5.1.1 认证鉴权实现

登录后签发 JWT；请求过滤器解析 token 并写入认证上下文；敏感接口通过角色注解限制。系统同时支持个人资料、密码修改、登出等认证相关操作。

### 5.1.2 业务模块实现

业务模块采用标准控制器-服务-数据访问层实现，支持分页查询、条件过滤、状态更新、统计汇总。  
例如考勤模块在服务层处理打卡规则、工时计算、异常判定，避免规则分散在前端。

### 5.1.3 治理模块实现

规则管理与模型管理模块提供参数化能力，使预警逻辑可维护；报表任务模块提供任务执行与结果留痕；消息模块负责协同通知。

## 5.2 前端实现

### 5.2.1 路由与权限

前端在路由守卫中校验登录态，在菜单层根据角色动态显示功能项。该策略既保证安全，也提升操作易用性。

### 5.2.2 可视化实现

分析看板、预警页面、专题分析页面采用卡片 + 表格 + 图表组合，实现指标概览、明细下钻、筛选联动和导出操作。

### 5.2.3 请求封装

统一 axios 实例注入 token，统一处理 401/403/404/500 异常，减少重复代码并增强交互一致性。

## 5.3 数据同步与分析实现

系统通过定时任务驱动数据同步与计算，分析服务按主题接口输出结果。  
在兼容模式下，分析链路仍可运行，确保答辩环境可演示。

## 5.4 预警与报表实现

预警服务通过规则阈值与模型参数联合计算风险结果。  
报表服务支持手工导出与任务调度，执行结果写入日志表，支持后续核查和重建。

## 5.5 部署实现

系统支持脚本化部署：构建、上传、远程执行、服务重启、Nginx 重载一体化。  
该流程显著降低了“本地可跑、线上不可用”的交付风险。

## 5.6 本章小结

本章给出了系统落地实现过程，证明设计方案具备工程可实现性。
